# UI — `main.ts`, Hotkeys, URL Flags, Save/Load

There is no UI framework. The entire DOM is one giant `app.innerHTML = …` template (`src/main.ts:85`), then queries refs and wires handlers procedurally. Every frame, `frame(now)` (`main.ts:3890`) ticks the sim, renders to canvas, then refreshes the DOM at most every 125 ms (`UI_REFRESH_INTERVAL_MS = 125` `main.ts:3862`).

**`main.ts` is 4349 lines.** The sole `export` is `buildDevTierOverlayString` (`main.ts:1890`) — used by harness assertions.

The `innerHTML` template lays out:

- **Top toolbar** (`main.ts:85`–~210) — Save/Load, Map Expansion, Play/Pause/Speed, view toggles (zones, service-nodes, inventory-overlay, glow, sprites, sprite-fallback), sprite status, autosave label.
- **HUD status strip** (`main.ts:95`–104; populated by `refreshHudStatus` `main.ts:1237`) — Crew/Oxygen/Power/Water/Food/Morale/Credits/Supplies with traffic-light coloring. Cycle/Day clock at `main.ts:1259`–1265.

- **Quest bar** — `renderQuestBar` painted into `#quest-bar`. Refreshed in `refreshUnlockLegendAndHotkeys` (`main.ts:1198`).
- **Right sidebar** — `refreshOpsModal`, `refreshSelectionSummary`, `refreshAlertPanel` (`main.ts:1825`), `refreshTierChecklist`, `refreshProgressionModal` (`main.ts:1995`), `refreshTrafficStatus`, `refreshTransportUi`, `refreshMarketUi`, `refreshPriorityUi`, `refreshExpansionUi`.
- **Modals** — Save, Market, Expansion, Progression, Priority, Dock (`refreshDockModal` `main.ts:2678`), Room (`refreshRoomModal` `main.ts:2715`), Agent (`refreshAgentModal` `main.ts:2871`). All wired via `wireModal` (`main.ts:3488`).

### Schema

`StationSnapshotV1` at `src/sim/save.ts:51`. Captures grid arrays (tiles/zones/rooms/policies), modules (sorted), dockConfigs, resources (credits/water/airQuality/legacyMaterialStock), `inventoryByTile` (per-node items), controls subset (shipsPerCycle, taxRate), unlocks (tier/ids/timestamps), progression lifetime counters incl. `archetypesEverSeen`. Envelope wraps with `schemaVersion = 2`, `gameVersion`, `createdAt`, `name`.

### Migration

- `parseAndMigrateSave` (`save.ts:557`) — accepts v1 envelopes and bare-tiles legacy snapshots (wraps them); collects warnings.
- `normalizeSnapshot` (`save.ts:299`) — defensive parse with per-field warnings; auto-elevates unlock tier if saved content (industrial/military ships, T2+ rooms/modules) requires more than declared (`requiredUnlockTierForSnapshotContent` `save.ts:185`).
- `hydrateStateFromSave` (`save.ts:688`) — calls `createInitialState`, replays grid arrays, places modules via `tryPlaceModule`, applies dock configs, pours item-stocks (clamping to capacity), resets transient state, ticks once paused.

### Player framing

Save modal opens from top-right (`#open-save-modal` `main.ts:106`). Supports create/quicksave/load/delete/download/import. Quicksave uses fixed slot id `quicksave` (sorts to top). Autosave toast: `Autosaved HH:MM` (`main.ts:4198`).

## Refresh cadence

Sim runs at frame rate; DOM at 8 Hz. Hover-tile diagnostic queries (`getRoomDiagnosticAt`) only every 250 ms (`HOVER_DIAGNOSTIC_REFRESH_INTERVAL_MS`). The decoupling is why heavy DOM work doesn't tank tick performance.

The Overlays palette owns diagnostic heatmaps. `Route Pressure` uses current actor paths to show visitor/resident/crew/logistics pressure and mixed-use conflicts; it is paired with the selected-crew route line so players can inspect both the station-wide pressure and one actor's immediate route.

## Known UX gaps (from playtest 2026-04-27)

- **Alerts panel does NOT aggregate per-room "inactive reasons"** — `refreshAlertPanel` (`main.ts:1825`) surfaces meal-stock-low / hull-leaks / power-deficit / open-incidents / critical-staffing, but does NOT loop over inactive rooms and surface reasons like "missing door / below minimum size / missing required modules" except via `topRoomWarnings` (limited to top 1–3). A new player painting a Cafeteria can sit there forever wondering why it doesn't activate. Diagnostic info is only available by clicking each room and reading the modal's `Inactive reasons:` line. Worth adding an aggregate "Why your rooms aren't active" alerts row.
- **Tier-up flash fires on visitor *arrival* (T1), not on visitor *served*.** The T1 unlock predicate (`archetypesServedLifetime ≥ 1`) increments based on `archetypesEverSeen` (a boolean per archetype, set on visitor SPAWN — not on visitor service success). Result: a station that can't actually serve anyone still gets a "Tier 1 unlocked!" celebration when its first visitor arrives. The flash is a false positive for "your station is working." Reconsider the T1 trigger — should ideally be on first successful meal exit, not first arrival.

## Trip-wires

- **`buildDevTierOverlayString` is the only export from `main.ts`.** Used by harness assertions; do not delete.
- **`applyHydratedState` (`main.ts:4154`) does `Object.assign(state, nextState)`** — relies on `state` being the same outer reference. Never replace it.
- **Mutating sim state from `main.ts` goes through the explicit barrel `src/sim/index.ts`** — when adding a new mutator, also add to that barrel or the import will break.
- **Pre-v2 saves are missing `progression`** — `archetypesEverSeen` defaults empty, which can permanently stick T2 if someone loads that save expecting in-progress visits to count. The migration path (`save.ts:716`–728) silently drops v1 ids (`tier1_stability`, `tier2_logistics`, `tier3_civic`); don't reuse those id strings.
- **`ItemNode.capacity` is determined by current `MODULE_DEFINITIONS`.** Changing a module's `itemNodeCapacity` clamps items at save-load time — old saves emit warnings.
- **`controls.spritePipeline` is a single-element union** (`'nano-banana'`). Adding a pipeline requires touching the type, the manifest router (`manifestPathForPipeline` `sprite-atlas.ts:138`), and any save migration.
- **The DOM template is one big string in `main.ts:85`.** Adding a section means editing that string; there's no component model. Keep the pattern.
