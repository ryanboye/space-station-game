# Roadmap Part 1 - Living Actors, Jobs, Logistics, and Residents

Last updated: 2026-05-02

How to use this plan: check off implementation items as they land, and append dated notes under each checkpoint whenever behavior, balance, UI, tests, or known debt changes. Keep browser/playtest notes unchecked until someone actually plays the build and records what happened.

Part 1 is the foundational simulation refactor. It should make the current game feel more alive before we add deeper utilities, sanitation, command, contracts, or advanced incidents.

The target outcome: visitors, residents, and crew interact with concrete objects; services use reservations instead of bunching; food and materials move in useful batches; production becomes visible module work; roles guide assignment without rigidly locking every task; residents begin to feel like citizens with routines and work.

## Why This Comes First

The current sim already has many systems, but they often resolve as global counters, room presence, or tiny one-off jobs. Adding sanitation, deeper utilities, station contracts, or patient dispatch before fixing object use and job assignment would amplify the same pain: more actors retargeting, more one-item jobs, more hidden causes, and more pressure on `src/sim/sim.ts`.

Part 1 builds the common spine:

- reservations,
- service providers,
- stock rules,
- batched logistics,
- job-board facade,
- task execution templates,
- role suitability,
- resident work/routines.

## Implementation Principles

- Do not rewrite every actor loop at once.
- Add new primitives beside existing systems, then migrate one vertical slice at a time.
- Every new API/state concept must have a game manifestation: inspector row, panel, overlay, alert, route, job marker, module action, or visible actor state.
- Prefer helper modules and small facades over growing the large sim file.
- Preserve old behavior until a migrated slice has tests and visible debugging.
- Use deterministic sim scenarios for logic and a browser/playtest note for user-facing behavior.

## Player-Facing Surfaces Needed In Part 1

Part 1 is not just backend architecture. The player needs tools to understand and use the new systems.

### Agent Inspector Upgrades

Every selected actor should expose:

- population: crew, visitor, resident;
- role/archetype;
- current desire: eat, shop, rest, work, haul, repair, socialize, exit, etc.;
- current task/activity: walking to provider, queued, using object, hauling batch, cooking, tending, restocking;
- target provider/module/room;
- reservation state: service slot, seat/use slot, item source, item target, job;
- blocked reason if stalled;
- route exposure summary when relevant;
- for workers: why this task was chosen, at least as a compact bid/suitability string later.

### Room Inspector Upgrades

Rooms should show:

- providers in the room: modules, capacity, users, queued actors, reserved slots;
- stock targets and current stock for item-node rooms;
- open/blocked jobs generated by the room;
- top blocked reason: no input, no output capacity, no worker, no path, no power/water/air, reservation conflict;
- throughput: served/min, produced/min, hauled/min where relevant;
- service quality: queue wait, seat shortage, stock shortage, bad route exposure.

### Module Inspector / Hover Detail

If the existing room inspector is the only click surface, module details can be nested there at first. Eventually modules should be inspectable because Part 1 makes modules the real service/work objects.

Module detail should show:

- provider type and capacity;
- current user and reservations;
- service tile;
- item stock/capacity if it has an item node;
- stock target;
- work task state if it publishes tasks;
- blocked reason.

### Station Ops Panel: Jobs Tab

Add or evolve an ops tab for jobs:

- open jobs by type: haul, build, repair, extinguish, cook, harvest, restock, fabricate, clean later;
- assigned jobs;
- blocked jobs by reason;
- average job age;
- stale/expired jobs;
- average batch size for logistics;
- scheduler health: jobs scanned, assignments made, workers idle by reason.

This can start as text/metrics and become richer later.

### Station Ops Panel: Logistics Tab

Logistics needs a readable surface before it gets more rules:

- food chain: Hydroponics rawMeal -> Kitchen meal -> Cafeteria serving stock;
- material chain: Intake/Storage -> Workshop/Construction;
- market chain: Workshop -> Market;
- each chain: source stock, target stock, desired stock, pending batches, blocked reason;
- job-miles/min and average batch size;
- top route-pressure conflict for logistics.

### Provider/Service Diagnostic Overlay

Add after the first provider migration, not necessarily at the very start.

Overlay should highlight:

- service providers by status: available, reserved, in use, blocked;
- queue pressure around serving stations/tables/beds/hygiene/leisure;
- provider shortages in red/orange;
- hover readout: provider type, queue, reservations, blocked reason.

### Job Markers And Routes

Jobs should show enough in-world feedback to avoid feeling abstract:

- marker at target module/site;
- color/status for open, assigned, blocked;
- route line for selected worker/job;
- item pickup/drop markers for logistics batches.

### Build/Palette Implications

Part 1 may need new or clarified objects:

- **Toilet**: strongly recommended once providers split hygiene/toilet. Current Sink/Shower abstraction works short term, but toilet as an object makes needs legible.
- **WaterFountain**: already exists in code; can become a drink provider.
- **Bench/Couch**: sitting/social provider.
- **VendingMachine**: snack/drink/shop provider with spend/trash hooks.
- **Stove/GrowStation/Workbench/ServingStation**: become work/stock providers, not just passive modules.
- **Resident business modules** later in Part 1: small shop counter, resident stall, or leased market/cantina variants if existing MarketStall/BarCounter are not enough.

Avoid adding many new modules before the provider model is working. Prefer migrating existing modules first.

## Data And System Shape

This is intentionally conceptual; exact types should follow existing code style.

### Reservation

Core fields:

- id;
- owner kind/id: actor, job, room, system;
- kind: provider slot, service tile, seat/use slot, source item amount, target item capacity, job assignment;
- target: module id, tile, item node id, job id, provider id;
- amount/capacity where relevant;
- createdAt, expiresAt;
- status/release reason.

Rules:

- all reservations expire;
- every task failure releases its reservations;
- item reservations must survive pathing delay but release on stale job;
- inspectors must be able to explain them.

### Provider

Core fields:

- provider id;
- provider kind: meal pickup, seat, vending, bed, shower, toilet, drink, leisure, workbench, stove, grow station, market, safety, medical later;
- module id and room anchor;
- service tile;
- capacity and reservation limit;
- current users;
- queue;
- cooldown/reuse timing;
- blocked reason;
- quality/status modifiers from room environment.

Providers should be derived from modules/rooms where possible, not hand-authored per actor.

### Task Template

Task templates should make actor behavior modular:

- `eat_meal`: reserve serving provider -> walk -> acquire meal -> reserve seat -> walk -> sit/eat -> pay/update need -> release.
- `use_vending`: reserve vending -> walk -> use -> spend/snack/drink -> maybe trash -> release.
- `rest`: reserve bed/seat -> walk -> use for duration -> recover -> release.
- `cook_batch`: reserve stove/service tile -> reserve raw input -> work timer -> reserve output -> produce meals -> release.
- `haul_batch`: reserve source amount -> reserve target capacity -> walk/pickup/drop -> release.

The task executor needs primitive steps:

- move to tile;
- wait/queue;
- use module for duration;
- pickup/drop item;
- update need/resource/economy;
- fail and release.

### Job Board Facade

Do not replace all current jobs immediately. First, wrap them:

- job type;
- source and target;
- priority;
- state;
- assigned actor;
- blocked reason;
- reservations;
- age/stale timer;
- player-facing label.

Then migrate execution job type by job type.

### Role Suitability

Roles should bias bids, not hard-lock the whole sim.

Crew role candidates:

- generalist;
- engineer/mechanic;
- logistics;
- cook/service;
- security;
- medic;
- pilot/operator.

Resident role candidates:

- resident worker;
- shopkeeper/vendor;
- bartender/service worker;
- hydro worker;
- workshop artisan;
- clinic aide;
- civic watch;
- unemployed/retired/student flavor later.

Visitors keep archetypes:

- diner;
- shopper;
- lounger;
- rusher;
- later: patient, inspector, contractor, VIP.

## Stop And Assess Before Part 2

Do not start Roadmap Part 2 until Part 1 has been played enough to answer:

- Are reservations reducing bunching without creating mysterious blocked states?
- Does the player understand providers and jobs from the UI?
- Is food logistics meaningfully better and less noisy?
- Are production tasks fun and legible, or do they feel like extra ceremony?
- Do roles make crew clearer without making emergencies brittle?
- Do residents feel more alive, or are they still just need bars walking around?
- Has `sim.ts` been contained, or do we need a module extraction pass before utilities?
