Starter State Refactor

space-station-game · design doc v3 · draft 2026-04-23 · author: seb, requested by BMO · v3 locks the Reactor-only pre-activation pivot (defer LifeSupport to T1 player task) per tinyclaw's layout analysis + airQuality decay check

TL;DR

Tinyclaw's fire 3/10 run 2 (option B: starter food chain pre-place) discovered a real scope ceiling: pre-placing rooms in createInitialState is not single-PR work because activating a room requires firing the same pipeline that normally only runs inside the live tick loop (size checks, module requirements, pressurization flood-fill, staff assignment, job-queue priming). She aborted her attempt cleanly and flagged the finding; BMO suggested a design doc.

This doc breaks the refactor into 4 sequenced PRs, plus a cleanup PR-5 (main.ts dock-placement fold) and a conditional PR-6 (scenario-fixture audit if >10 fixtures affected). Each is reviewable on its own, leaves the game playable after merge, and sets up the next. v3 pivot (2026-04-23): PR-2 drops LifeSupport pre-placement — LifeSupport becomes the player's first T1 task. Reactor alone at cold start + finite air buffer that T1 must fire before consuming. Tighter scope for PR-2 (~100-150 LoC instead of ~150-200), cleaner pacing, LifeSupport-adjacency question resolves for free. Total estimated scope: 2-3 days of focused work by one claw plus review cycles. Gated on awfml's tutorial-pacing answer (the right end state depends on what T0 should feel like for a first-time player).

Invariant preserved across all 4 PRs: createInitialState({ seed }) returns a deterministic, tick-ready StationState. No async. No I/O. The starter layout is pure functional construction — same seed always produces the same station. Reproducibility is non-negotiable because save/load + scenario fixtures + harness tests all depend on it.

Problem statement

Today createInitialState (src/sim/sim.ts:6869) allocates the arrays and paints:

  • A 10×10 floor patch at (25,14)..(34,23) with wall border
  • A 3×3 reactor core at the grid center with doors on cardinals
  • Lane profiles + empty module/incident/visitor/resident/crew arrays

Then PR #20 added a tile-level post-step in main.ts: two setTile(…, TileType.Dock) calls at the east hull to auto-create a visitor dock via rebuildDockEntities. That's the full current starter state.

What's missing for a "playable demo on arrival" station:

  1. Reactor is painted as Tile+Room but NOT pressurized, NOT powered. Air fails, crew suffocate if they spawn.
  2. LifeSupport isn't placed at all. Players have to discover + zone + build it before anything survives.
  3. No food chain. Visitors arrive, can't eat, leave unfulfilled. Negative review spiral starts in the first 2 minutes of play.
  4. No crew. ensureCrewPopulation lazily spawns them on first tick, but their jobs queue is empty so they idle.

Tinyclaw's abandoned attempt tried to solve (3) directly and hit the activation-pipeline wall: paintRoom + placeModuleOrThrow wires the tiles, but the room isn't active until pressurization + min-tile + module-requirement checks pass. Those checks currently run inside the tick loop via buildHabitat + rest of tick(), not from a cold start. So the only path today is: cold-start → first tick fires → room gates get evaluated → rooms light up. We want to shift some of that to cold-start time.

4-PR breakdown

PR-1Activation pipeline cold-start hooks

scope: refactor-only, no behavior change · ~200-300 LoC · sim-test coverage required

Extract the per-tick activation sequencing from tick() into a callable runActivationPipeline(state) that createInitialState can invoke at the end of its own construction. The callable runs, idempotently:

  • bumpTopologyVersion(state) — forces cache invalidation
  • rebuildRoomClusters(state) — maps rooms → cluster ids
  • recomputePressurization(state) — BFS flood-fill from sealed-room pressure sources
  • rebuildDockEntities(state) — already exists, called on dock tile changes
  • evaluateRoomActivation(state) — checks min-tiles + required-module per room type, sets roomActiveByTile[tile] flags
  • rebuildJobQueue(state) — drains completed jobs, emits new jobs for active rooms

This PR does NOT change what happens at runtime. It just factors the existing in-tick call sequence into a named function. Tick still calls it on topology changes; createInitialState doesn't call it yet (that's PR-2).

Acceptance: all existing sim-tests pass. Add one new test: testRunActivationPipelineIdempotent — call it twice in a row on a quiescent state, assert the state is deep-equal via sorted-keys JSON serialization (Sets/Maps don't serialize deterministically otherwise; per BMO's v2 note). Alternative bar if that trips on hidden non-determinism: "all state.metrics + all actor positions + all state.tiles entries identical" — same regression coverage, fewer false negatives.

PR-2Pre-activate Reactor + starting air buffer (LifeSupport deferred to T1)

scope: behavior change · ~100-150 LoC · sim-test + scenario-fixture coverage required

Pivot from v1/v2 (tinyclaw's #54 layout analysis): LifeSupport's 6-tile footprint doesn't fit cleanly in the 10×10 interior alongside Kitchen + Hydroponics + Cafeteria + Reactor. Instead of scope-blasting the interior to 12×10 or forcing an L-shaped LifeSupport, defer LifeSupport to the player's first T1 task. Reactor alone satisfies the cold-start survival invariant — for a bounded air-buffer window, calibrated so T1 unlocks before airQuality hits the distress threshold.

Extend createInitialState to:

  1. Ensure Reactor room's required module is placed at the core-center tile (confirm the exact ModuleType name against balance.ts when writing the PR — the v1 doc speculated ModuleType.Reactor; verify).
  2. Initialize state.metrics.airQuality = 100 (already the default; verify not regressed). This is the buffer that keeps crew alive until the player places LifeSupport.
  3. Do NOT place LifeSupport. It becomes the player's T1 onboarding task: station breathes initially, air quality visibly drops over T0, unlocking "build LifeSupport" as the T1 progression moment.
  4. Call the new runActivationPipeline(state) before returning — Reactor activates on geometry alone since its 3×3 frame at x=29..31, y=18..20 (coreY=19) already satisfies its predicate.

Air-buffer calibration (per tinyclaw's sim.ts:6171 reading): airQuality decays as (airSupply - airDemand) * dt * 1.7. With no LifeSupport, airSupply = 0 and the rate reduces to -airDemand * dt * 1.7. The window between airQuality = 100 and the distress threshold (~35) is what PR-2 + PR-4's T1-unlock timing must cooperate to span. Calibration is awfml's pacing call: the T1 unlock trigger must fire BEFORE the buffer runs out. tinyclaw to confirm airDemand values per crew and derive the budget during PR-2.

Result: cold-start state has power + sealed rooms + a finite air buffer. Crew survive the tutorial window, player builds LifeSupport at T1, air stabilizes. Pacing: "watch the station tick" → "uh oh air quality dropping" → "T1 unlock: build LifeSupport" — classic onboarding beat.

Acceptance: three sim-tests —

  • testColdStartReactorActiveOnReturn — assert Reactor is Active immediately after createInitialState returns (not after first tick).
  • testColdStartAirDecaysWithoutLifeSupport — run 60s, assert airQuality strictly decreasing (confirms the decay model actually fires; guards against a regression where airSupply accidentally defaults nonzero).
  • testColdStartAirBufferExceedsT1UnlockTime — run until T1 unlocks (per awfml's locked pacing rule), assert airQuality > 35 for the full duration. If T1 doesn't unlock in time, test fails loudly — tuning is awfml's call then.

Scenario fixture cold-start-reactor-only.json: zero player action, run until the air buffer drops below 35 (not a fixed tick count — runtime-bounded by the decay curve), assert zero crew deaths within that window. Fixture serves as a regression guard for the T0→T1 timing contract.

PR-3Pre-place food chain (Hydroponics + Kitchen + Cafeteria)

scope: behavior change · ~250-350 LoC · sim-test + e2e coverage required

Paint 3 more rooms + their minimum-required modules + appropriate wall/door connectivity. Coords locked by tinyclaw's layout analysis (PR #54 comment):

  • Kitchen — x=25..28, y=14..15 (4×2 = 8 tiles, meets minTiles=8). Place ModuleType.Stove (2×1) + ModuleType.ServingStation (2×1) — both fit.
  • Hydroponics — x=25..28, y=21..22 (4×2 = 8 tiles, meets min 8). Place ModuleType.GrowStation (2×2) — fits with spare tile.
  • Cafeteria — x=32..34, y=14..17 (3×4 = 12 tiles, meets min 12). Place ModuleType.Table (2×2) — fits with spare floor for pathing.

Corridors + connectivity: left column x=25..28, y=16..20 + right column x=32..34, y=18..23 path-connect all rooms to the Reactor core doors. Doors on each room's edge facing the corridor. Verify via runActivationPipeline at the end of createInitialState.

PR-4Seed crew + initial jobs

scope: behavior change · ~100-150 LoC · sim-test + e2e coverage required

Extend createInitialState to:

  1. Populate state.crewMembers with 3 crew (tinyclaw confirmed 2026-04-23: "8 is payroll-pressure + information overload for a tutorial; 3 shows what crew does without swamping the player. hire up naturally"). Replaces the current lazy-on-first-tick ensureCrewPopulation cap of 8. Spawn at designated tiles near the reactor core, default healthState: 'healthy', no pre-set Phase-5 condition state so the upcoming spec stays clean.
  2. Seed state.jobs with the initial set that rebuildJobQueue would produce after PR-3's rooms activate: "staff kitchen", "staff cafeteria", "staff hydroponics". Crew pick these up on first tick instead of idling.
  3. Update the lazy-populate path in ensureCrewPopulation so it NO-OPs if state.crewMembers.length > 0 on entry. Backwards-compat for any path that still expects lazy spawn.

Player-visible state delta heads-up (per BMO's v2 note #3): the 8→3 crew change is not purely internal — it shows in the crew list sidebar, the payroll metric, and any sim-metric widget that counts actors. Scenario fixtures that reference crew.total === 8 or count crew actors will need updates. Rolls into Risk #4's audit (flagged below as potentially PR-sized).

Result: first tick has 3 crew walking to assigned jobs. First visitor can arrive and get served. The tutorial's opening beat is "watch the station breathe" rather than "figure out why nothing's happening."

Acceptance: testColdStartCrewAssignedJobsOnFirstTick — assert crew[0..2] have assignedJobId !== null after tick(state, 0.01). e2e test: no player action for 5 minutes, assert T0→T1 progression fires (per awfml's pacing answer — either at first-meal-served or first-visitor-served or whatever the locked spec says).

Risks

  • Air-buffer duration < T1-unlock trigger time. Since PR-2 defers LifeSupport to a T1 player task, airQuality decay is on a finite clock from cold start. If the T1 unlock predicate (awfml's pacing call) doesn't fire before airQuality hits the distress threshold (~35 per the current tuning), crew start dying during the tutorial. Mitigation: testColdStartAirBufferExceedsT1UnlockTime in PR-2 is the regression guard; if it fails, either the buffer gets bumped (start airQuality higher, or reduce decay rate for T0 only) or awfml's pacing rule gets tightened.
  • Tutorial-pacing coupling. If awfml picks "T0 must hold for 30s regardless of event," PR-4's crew-job-seed + PR-3's food-chain-activation might fire T0→T1 faster than intended. Mitigation: add a `minTierDwellSec` gate in the unlock predicate, configurable. Pairs with the air-buffer risk above — both point at the same "what does T0 feel like" answer from awfml.
  • Save-format migration. Old saves pre-refactor expect lazy crew spawning. hydrateStateFromSave must NOT call runActivationPipeline (saves are already-activated snapshots). Enforce with a separate createPreActivatedState / hydrateToActiveState split if needed.
  • Determinism. Adding rooms + crew to the starter state expands the seed's effect surface. Bit-identical save-roundtrip tests must pass with the new starter before merging PR-2+.
  • Scenario fixtures currently assume a bare starter. ghost-town.json, isolation.json, etc. need audit — any that assumed "zero rooms + zero crew" as their baseline need explicit teardown in their layout section. Per BMO's v2 note #5: if the audit uncovers 10+ affected fixtures, the update may itself be PR-sized — split off as PR-6 with its own review burden rather than bundling into PR-2/3/4.

Non-goals (punt)

  • Toolbar integration for "reset to starter" button (already covered by BMO's HUD rework in PR #41 + #43)
  • Multi-seed variety — starter state is single-canonical for v1. Randomized starters can be a follow-up once the single variant is tuned.
  • Phase 5 health state pre-seeding (crew default healthy; Phase 5 spec owns the richer state machine)
  • Visitor pre-seeding — visitors still spawn via dock ship arrival, same as today. No pre-placed visitors at cold start.

Rollout

  1. Ship PR-1 first — refactor only, zero behavior change. Validates the activation-pipeline-extraction shape.
  2. Gate PR-2 merge on awfml's tutorial-pacing answer (to avoid redoing the T0-timing calibration later).
  3. PR-3 + PR-4 can ship independently after PR-2, in either order. PR-4 depends on PR-3 only for the jobs seed (if PR-3 hasn't shipped, PR-4 skips the food-chain jobs + only seeds crew).
  4. PR-5 — `main.ts` cleanup (promoted from trailing bullet per BMO's v2 note #4). Remove PR #20's post-createInitialState setTile dock-placement from main.ts — fold into createInitialState itself for consistency. Its own PR with its own test burden: verify PR #20's existing dock behavior doesn't regress (spawn visitor, confirm dock assignment + approach-tile pathing works unchanged).
  5. PR-6 — scenario-fixture audit (conditional, per BMO's v2 note #5). If Risk #4's audit finds >N fixtures affected (rough threshold: 10), carve out a dedicated cleanup PR rather than bloating PR-2/3/4 reviews with fixture updates. If the audit is small (< 10 fixtures), inline into the relevant behavior-change PR.
